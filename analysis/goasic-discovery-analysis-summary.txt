Key findings for parent agent:

3) Actual discovery flow in manager.go
- `Start()` launches:
  - `scanLoop(ctx)` for periodic subnet scans
  - `dataCollectionLoop(ctx)` for periodic polling of already-known miners
- `scanLoop()`:
  - runs one immediate `ScanNetwork(ctx)`
  - then repeats every `time.Duration(m.cfg.GoASICScanInterval) * time.Second`
- `ScanNetwork()`:
  - uses `scanSubnetFn(ctx, m.cfg.GoASICNetworkRange, 50)`
  - `scanSubnetFn` defaults to `goasic.ScanSubnet`
  - returned miners are `[]goasic.Miner`
  - for each miner:
    - calls `miner.GetData(ctx)`
    - on success maps to `DiscoveredDevice`
    - on failure still adds/updates a device using:
      - `IP: miner.IP()`
      - `Model: "Unknown"`
      - `Make: "Unknown"`
      - `Status: StatusError`

4) Duplicate handling
- Manager stores miners in `map[string]*DiscoveredDevice`
- Uniqueness key is IP
- `addOrUpdateDevice()` updates an existing entry when the same IP is seen again
- For dashboard `refreshData`, IP is the correct dedupe key

5) Live refresh flow
- `dataCollectionLoop()` runs every 30s
- `collectAllDeviceData()` polls all current devices concurrently
- `collectDeviceData(ctx, device)`:
  - calls `detectMinerFn(ctx, device.IP)`; default is `goasic.Detect`
  - then `miner.GetData(ctx)`
  - then updates existing device:
    - `Data = convertMinerDataToDeviceInfo(data)`
    - `LastSeen = time.Now()`
    - `Status = StatusOnline`
  - triggers `onDeviceUpdate`

6) Callback behavior
- `OnDeviceDiscovered`: called only when a new IP is first inserted
- `OnDeviceUpdate`: called on updates and on live poll refresh
- `OnError`: called for subnet scan / detect / get-data failures
- `OnDeviceLost`: setter exists but not invoked anywhere in `manager.go`

11) Nil-handling caveat inferred from manager usage
- In `GenerateDeviceReports()` these `device.Data` fields are treated as optional/pointer-like:
  - `Hashrate`
  - `Temperature`
  - `Wattage`
  - `Efficiency`
  - `Uptime`
- Parent agent should normalize missing values before CSV writes and before recalculating dashboard stats

12) Limits / blockers
- I did not modify source files
- I could not directly inspect `models.DeviceInfo`, `models.DeviceReport`, or `config.Config` definitions from the available files, so exact pointer/value types are inferred from usage in `manager.go`
- Only pool1 fields are explicitly mapped (`Pool1URL`, `Pool1User`); no pool2/pool3 mapping is present in current manager conversion
- No explicit absent-miner offline marking is implemented during rescans; manager dedupes by IP and updates known devices, but `OnDeviceLost` is currently unused
